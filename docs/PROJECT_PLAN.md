# CertVault 项目规划

## 一、产品定位

CertVault 的目标不是堆叠 Apple API 功能，而是把高风险、步骤多、容易过期的开发者资源管理流程，做成可追踪、可恢复、可审计的一套工作台。

核心用户场景：

1. 导入并验证 Apple 开发者账号。
2. 管理设备、证书、Bundle ID 与描述文件。
3. 完成设备一键绑定和资源打包下载。
4. 管理 TestFlight 测试组与公开邀请。
5. 配置并验证 APNs 推送能力。
6. 在 Web 和 iOS 端查看状态、异常与到期提醒。

## 二、系统边界

```text
Web 管理后台 ─┐
              ├── REST API ── PostgreSQL
iOS / Widget ─┘       │
                      ├── Apple Developer / App Store Connect
                      ├── APNs / SMTP
                      └── 本地文件或 S3/MinIO
```

- Web 与 iOS 共享同一套业务 API，不在客户端复制核心业务规则。
- 后端负责 Apple 凭据、证书私钥、同步任务、权限与审计。
- 数据库保存业务状态；大文件和敏感文件通过独立存储层管理。
- 公开 UDID/TestFlight 页面与登录后台采用明确分离的鉴权边界。

## 三、近期优先级

### P0：安全与可维护性基线

- 轮换历史中出现过的服务器、数据库、SSL 与应用凭据。
- 建立数据库迁移机制，停止在启动流程中临时修改表结构。
- 统一配置校验：缺少关键环境变量时直接拒绝启动。
- 为账号、证书、描述文件、推送等敏感操作增加审计日志。
- 增加最小 CI：前端构建、后端语法检查、关键测试、敏感信息扫描。

完成标准：新环境能根据文档部署；敏感信息不进入仓库；每次发布可回滚。

### P1：后端稳定性

- 将 `app.js` 中的业务逻辑下沉到 route/service/repository 分层。
- 统一 API 响应、错误码、日志格式与请求追踪 ID。
- 为 Apple API 调用增加超时、限流、重试和可识别的失败状态。
- 将耗时同步、证书生成和批量操作改为后台任务。
- 覆盖登录、数据隔离、证书创建、设备绑定、TestFlight 邀请等核心测试。

完成标准：关键流程有自动化测试，外部 API 短暂失败不会造成重复数据或未知状态。

### P2：产品体验统一

- 统一 Web 与 iOS 的状态命名、错误说明和操作反馈。
- 拆分 Web 公共依赖与页面包，压缩首屏 JavaScript 和图片体积。
- 建立“资源健康中心”：到期、失效、配额、同步异常集中展示。
- 把一键绑定拆成可观察的步骤，支持失败重试与结果下载。
- 完善 TestFlight 邀请生命周期、名额和操作记录。
- 明确超级管理员与普通用户的权限矩阵。

完成标准：用户能理解当前状态、失败原因和下一步操作，不依赖人工查日志。

### P3：运营与规模化

- 增加监控指标、告警、任务队列和定时备份验证。
- 建立版本号、变更日志、Git 标签和发布审批流程。
- 支持对象存储生命周期、数据导出与账号注销。
- 根据真实使用量再评估多实例、缓存与队列方案。

完成标准：可持续发布、可定位故障、可验证备份、可审计变更。

## 四、推荐迭代节奏

每个迭代只选择一条可验收的业务主线，同时安排安全或测试债务：

1. 先写验收场景和 API 影响范围。
2. 后端实现与测试先稳定接口。
3. Web 与 iOS 按同一状态模型接入。
4. 在测试环境验证 Apple 真实链路。
5. 生成发布包、备份、上线、验证并记录版本。

功能完成必须同时满足：代码已合并、测试通过、文档更新、监控可见、具备回滚路径。

## 五、下一轮建议任务

1. 完成敏感凭据轮换并建立密钥清单。
2. 给后端增加环境变量启动校验和 `/api/status` 公共存活检查。
3. 引入数据库迁移目录并记录当前 schema 基线。
4. 为认证与用户数据隔离补第一批集成测试。
5. 将发布检查固化为 CI 和版本化 release。

在这五项完成前，建议控制新功能数量。这里像整理机房布线：先把标签和保险丝弄明白，之后加设备才不会每次都心惊肉跳。
